## 1. 全局业务规则（Cross-Cutting Business Rules）

以下规则约束所有模块，编号 `GBR-*`，实现时作为平台级中间件/服务强制执行，任何模块不得绕过。

### GBR-1 身份防火墙（Identity Firewall）

1. 任何对外接口（API 响应、页面、通知、导出文件、AI 生成内容）中，交易对手方只能以平台代码出现。禁止出现：公司全称、联系人姓名、电话、邮箱、详细地址、税号、公章、银行账户。
2. 可见性由统一的 **字段级可见性策略（Field Visibility Policy）** 控制，按"角色 × 字段 × 业务上下文（如订单是否确认）"三维判定，策略数据化存储、可配置，默认拒绝（deny-by-default）。
3. 初稿 §11.2 的可见性矩阵作为默认策略基线，并补充：订单确认后，双方可见**履约必需的最小信息**（收货地址、清关抬头），但仍不可见对方公司全称与联系方式。
4. AI 生成的任何文本（翻译、描述、报告、聊天回复）输出前必须经过 **PII 泄露过滤器**（正则 + NER 双通道），命中即阻断并审计。

### GBR-2 编码规则引擎（Code Generation Engine）

1. 所有公开业务编码由中心化编码服务签发，规则可配置（前缀、位数、序列、日期段），保证全局唯一、不可预测遍历步长（序列加随机跳步）。
2. 默认规则：供应商 `SP-{6位序列}`、采购商 `BY-{6位序列}`、产品 `PRD-{6位}`、订单 `ORD-{YYYYMMDD}-{5位}`、拍卖 `AUC-{YYYYMMDD}-{4位}`、RFQ `RFQ-{YYYYMMDD}-{4位}`、期货合约 `FUT-{YYYYMMDD}-{4位}`、发票 `INV-{YYYYMMDD}-{5位}`。
3. UUID 为数据库主键，永不出现在 URL、API 响应、日志摘要中；对外一律用公开编码或短哈希。

### GBR-3 国际化（i18n）

1. 支持 `zh-CN` / `en` / `fr` 三语，架构上支持任意增加语种。
2. UI 文案：前端 i18n 资源文件；**业务数据**（产品名/描述、等级名、品种名、单位、通知模板、邮件、发票、单证）：数据库翻译表（`entity_translations`），不用平铺列。
3. 内容缺失语种时按回退链渲染（fr → en → zh），并标记"机器翻译待复核"。
4. 数字/日期/货币/度量单位按 locale 格式化；重量统一存 kg，展示层换算。

### GBR-4 多币种

1. 支持 EUR / USD / CNY / GBP / JPY，基准币 EUR。
2. 汇率来源可配置（默认 ECB 日频），每笔报价/订单/发票固化当时汇率快照，事后汇率变动不影响已成交单据。
3. 所有金额 `DECIMAL(14,2)`（日元 0 位小数按 locale 处理），金额运算禁止浮点。

### GBR-5 审计（Audit Everything）

1. 关键写操作（登录、审核、上架、出价、下单、支付、状态迁移、配置变更）全部产生审计事件：操作者、角色、时间、IP、UA、目标实体、前后值 diff。
2. **读敏感信息也要审计**：任何真实身份字段的解密展示均记录（谁、何时、看了谁的哪个字段、事由），穿透查看走审批流（见模块 19）。
3. 审计日志只追加（append-only），普通管理员不可删改。

### GBR-6 状态机纪律

1. 订单、拍卖、报关、支付、争议、期货合约等生命周期实体一律用显式状态机：迁移表数据化定义（源状态、目标状态、允许角色、守卫条件、触发副作用/领域事件）。
2. 禁止直接 UPDATE 状态列；只能通过状态机服务迁移，每次迁移写审计 + 发领域事件。

### GBR-7 软删除与数据规范

全表统一：UUID 主键、`created_at/updated_at/created_by/updated_by/deleted_at/version`（乐观锁）。物理删除仅限 GDPR 删除请求流程。

## 2. 模块级功能需求（22 Modules）

每模块含：目标、功能需求（FR）、业务规则（BR）、实施期（P1/P2/P3）。

### M01 Marketplace（前台市场）— P1

**目标**：三语公开商城，Guest 可浏览脱敏市场，Buyer 可完成直购。

- FR-01-01 产品目录：按品类/品种/等级/规格/产地（国家级粒度）/价格区间筛选；供应商仅显示代码 + 认证徽章（"已验证供应商"，不显示证书原件）。
- FR-01-02 搜索：关键词全文检索（中/英/法分词）+ 语义搜索（pgvector）混合排序。
- FR-01-03 产品详情：三语描述、等级说明、阶梯价（登录 Buyer 可见）、库存状态（仅"充足/紧张/售罄"档位，不露具体数字）、溯源摘要（脱敏：产地水域类型、鱼龄、加工日期，不含基地名/企业名）。
- FR-01-04 收藏夹、购物车（跨供应商拆单）、比价。
- FR-01-05 SEO：三语 SSR、hreflang、结构化数据；oussouri.fr 默认 fr，oussouri.com 默认 en。
- BR-01-01 未登录可浏览目录与详情，但价格显示"登录询价"；下单/收藏/加购需 Buyer 认证通过。
- BR-01-02 所有浏览/搜索/收藏/加购行为写入行为事件流（M13 撮合数据源）。

### M02 Supplier ERP（供应商工作台）— P1 基础 / P2 溯源全量

**目标**：供应商自助管理"产源 → 加工 → 产品 → 库存 → 订单履约"。

- FR-02-01 入驻与资质：注册 → 提交营业执照、SC 证、CITES 资质、出口许可 → 管理员审核 → 激活。证书有过期日，到期前 30/7 天提醒，过期自动降级为"不可上架"。
- FR-02-02 溯源域（行业无关抽象）：产源单元（Production Unit：养殖场/池塘/车间/庄园/牧场）→ 原料批次（Source Batch：鱼群批次，含 RFID 区段、品种、数量、鱼龄）→ 个体资产（可选，RFID 个体鱼）→ 养护记录（投喂/健康/用药，含休药期校验）→ 加工批次（原料重、成品重、得率、工序步骤、质检结论）。鲟鱼字段组作为"行业模板"配置。
- FR-02-03 产品管理入口（复用 M04）、库存入口（复用 M05）、订单履约（确认、备货、上传运单）。
- FR-02-04 经营看板：在售 SKU、库存效期预警、订单漏斗、结算中金额。
- BR-02-01 加工批次必须关联原料批次方可生成可售库存（溯源链不断裂）；休药期未满的批次禁止进入加工。
- BR-02-02 供应商相互不可见；供应商不可见平台整体供需数据。

### M03 Buyer CRM（采购商工作台 + 平台侧客户管理）— P1 基础 / P2 完整

- FR-03-01 Buyer 自助端：注册审核（公司、税号 VAT、进口资质）、订单中心、单证下载（脱敏版）、收藏/RFQ/竞拍记录、收货地址簿。
- FR-03-02 平台侧 CRM（Broker/客服用）：客户 360 视图（脱敏默认 + 穿透审批）、分层标签（米其林餐厅/批发商/进口商）、活跃度与流失预警（如"30 天未下单"触发商机，联动 M13）、跟进记录（Timeline）。
- FR-03-03 LOI/采购意向书：Buyer 可发布结构化采购意向（品种、规格、数量、目标价、交期、目的港），作为撮合信号源。
- BR-03-01 Buyer 信用评分（0-100）：由按时付款率、争议率、累计采购额加权，规则可配置；影响期货保证金比例与账期政策。

### M04 Product Management（产品管理）— P1

- FR-04-01 产品建模：品类（可配置树）、品种（Species 字典，含拉丁名/CITES 附录）、等级（Grade 字典，平台自定义命名如"赫哲传承级"）、规格（50g/100g/250g 罐装等 SKU 变体）、HS 编码、MOQ、单位。
- FR-04-02 三语内容：名称/描述/卖点走翻译表；接入 M15 AI 翻译生成初稿 + 人工确认发布。
- FR-04-03 阶梯价（数量区间 × 币种 × 生效期）；价格变更留历史版本。
- FR-04-04 上架审核流：供应商提交 → 质检员/管理员审核（内容合规 + 溯源链完整）→ 上架。
- FR-04-05 AI 生成产品描述（M22 能力）：基于结构化参数生成三语文案，人工确认后生效。
- BR-04-01 产品必须绑定有效溯源加工批次与有效证书才能上架；证书过期自动下架并通知。
- BR-04-02 杂交品种用父本×母本双外键建模，公开代码用规范化编码（如 `DAUHUS`）。

### M05 Inventory（库存）— P1

- FR-05-01 批次库存：按"产品 SKU × 加工批次"维度管理，含生产日期、效期、存储温度、仓位。
- FR-05-02 库存事务：入库/预留（下单锁定）/释放（取消）/扣减（发货）/盘点调整，全部走库存流水账（Ledger），当前量 = 流水聚合，禁止直接改余额。
- FR-05-03 效期与温控告警：临期（如剩余保质期 <30%）告警并可自动触发促销/拍卖建议（联动 M13）。
- BR-05-01 预留有 TTL：意向单 24 小时未支付自动释放（联动 M14 居间锁货规则）。
- BR-05-02 可售量 = 在库 − 预留；超卖由数据库层约束兜底（事务 + 检查约束）。

### M06 Auction（拍卖）— P3

- FR-06-01 拍卖类型：英式（明拍加价）、荷兰式（降价拍）、密封投标；场次由供应商申请、平台审核排期。
- FR-06-02 保证金：参拍前按场次规则缴纳（Stripe 预授权或托管），未中标自动退还，中标转入货款。
- FR-06-03 实时竞价：WebSocket 推送、出价校验（≥当前价+加价幅度）、代理出价（设置上限自动跟价）、反狙击（结束前 N 分钟内出价自动延时 M 分钟）。
- FR-06-04 保留价不公开；未达保留价流拍；中标自动生成订单（AUCTION 类型）进入标准履约流。
- BR-06-01 出价人互相只见"竞买号"（场次内临时代号 B01/B02），连平台代码也不显示，防串拍。

### M07 RFQ（询价）— P2

- FR-07-01 Buyer 发布 RFQ（品种/等级/规格/数量/目的港/期望价/截止时间），可定向（指定供应商代码）或平台广播（由 Broker 分发）。
- FR-07-02 供应商提交报价（Quote：单价、MOQ、交期、有效期），多轮议价（版本化）。
- FR-07-03 Buyer 接受报价 → 自动生成订单；Broker 可全程介入调价（居间模式）。
- BR-07-01 RFQ 与 Quote 中的自由文本经过 PII 拦截（防联系方式夹带）。

### M08 Order Management（订单）— P1

- FR-08-01 订单类型：DIRECT / AUCTION / RFQ / FUTURES / BROKER（居间代下单）。
- FR-08-02 订单状态机（正常流）：`DRAFT → PLACED → PAID_ESCROW → CONFIRMED → PREPARING → SHIPPED → IN_CUSTOMS → CUSTOMS_CLEARED → DELIVERED → COMPLETED`；分支：`CANCELLED`（PAID 前可取消；PAID 后走退款流）、`DISPUTED`（DELIVERED 前后均可发起）→ `RESOLVED`。每次迁移记录操作者与时间。
- FR-08-03 单据自动生成：形式发票 PI、商业发票、装箱单（三语模板，Buyer 侧脱敏、平台抬头）。
- FR-08-04 签收确认：Buyer 主动确认，或送达后 N 天（默认 7，可配置）自动确认。
- FR-08-05 期货合约（P3）：供应商发布预售合约（预估产量/交割期/锁定价）→ Buyer 认购 → 保证金 10–20%（按信用评分浮动）→ T-30 交割提醒 → 确认可交割量 → 尾款 → 转标准订单。
- BR-08-01 订单金额、汇率、阶梯价在 PLACED 时刻固化快照。
- BR-08-02 一张订单单一供应商；购物车跨供应商自动拆单（对 Buyer 呈现为一个"采购批次"聚合视图）。

### M09 Escrow Payment（托管支付）— P1

- FR-09-01 Stripe Connect：供应商入驻时创建 Connected Account（KYC 由 Stripe 承担）；Buyer 支付进入平台 Stripe 账户（Separate Charges & Transfers），签收/自动确认后 Transfer 给供应商，平台佣金自动截留。
- FR-09-02 支付方式：卡、SEPA/银行转账（Stripe 渠道）；大额线下电汇走"手工对账入账"工作流（财务上传水单、核销）。
- FR-09-03 内部账本：双分录 Ledger（Buyer 在途、Escrow 冻结、平台佣金、供应商应收、退款），任何资金状态变化先记账后动作，日终与 Stripe 对账。
- FR-09-04 佣金规则引擎：按品类/供应商等级/订单类型配置费率；居间订单可叠加 Broker 服务费。
- FR-09-05 退款与争议：争议冻结自动释放计时；裁决结果驱动全额/部分退款（Stripe Refund / 反向 Transfer）。
- BR-09-01 平台不自持客户资金，全部经 Stripe 持牌通道；账本为影子账，以 Stripe 为资金事实源。
- BR-09-02 分账触发条件：`DELIVERED` 确认 + 无进行中争议 + 争议期（默认 48h）届满。

### M10 Logistics（物流）— P2

- FR-10-01 多段运输建模：一票物流含多段 Leg（如 HRB→PEK→CDG + 末端冷链配送），每段独立承运人/单号/状态。
- FR-10-02 冷链监控：温度日志上传（CSV/API），超阈值（如 >0℃）自动告警并存证；GPS 轨迹存档。
- FR-10-03 状态推送：关键节点（起运/到港/清关中/派送/签收）三语通知 Buyer 与供应商。
- BR-10-01 物流信息中双方地址按可见性策略裁剪（供应商见收货地址，不见 Buyer 公司全称）。

### M11 Customs（清关）— P2

- FR-11-01 出口/进口双报关单管理：单号、报关行、HS 编码、申报价值、关税/VAT、查验结果、状态机（`SUBMITTED → INSPECTION → CLEARED / REJECTED`）。
- FR-11-02 单证清单引擎：按"品类 × 出口国 × 进口国"配置必备单证模板（鲟鱼→法国默认 7 件：商业发票、CITES、原产地证、装箱单、空运单、卫生证、健康证），缺件不允许进入 SHIPPED。
- FR-11-03 CITES 管理：许可证号、有效期、**配额余量跟踪**（每证可用重量，随发货扣减），临期/超配额告警。
- FR-11-04 AI 清关助手（P3）：单证 OCR 提取、一致性交叉校验（发票金额↔报关价值↔订单金额）、目的国税费试算。
- BR-11-01 清关专员是唯一可见报关真实抬头的操作角色（穿透审计生效）。

### M12 Certificates（合规单证中心）— P2

- FR-12-01 统一单证库：主体资质（营业执照/SC/出口许可）、批次单证（CITES/健康证/卫生证/检测报告）、订单单证（发票/装箱单/运单），统一元数据（类型、编号、签发方、效期、关联实体）。
- FR-12-02 **脱敏发送**：Broker/客服勾选单证 → 系统自动遮盖供应商公章、企业名称、地址区域（预标注遮盖区 + AI 检测兜底）→ 加平台水印与唯一追踪码 → 生成 PDF 副本发送至 Buyer 站内档案。原件永不外发。
- FR-12-03 每份脱敏副本可追踪（谁生成、发给谁、何时、追踪码），防扩散溯源。
- BR-12-01 脱敏模板按单证类型预配置；新单证类型上传时强制人工标注遮盖区一次。

### M13 AI Matchmaking（智能撮合）— P2 规则版 / P3 模型版

**平台核心价值模块。**

- FR-13-01 行为事件流：搜索词、筛选条件、浏览时长、收藏、加购、RFQ、LOI、订单、流失信号（30 天未下单）统一埋点入事件表。
- FR-13-02 需求画像：按 Buyer 聚合出偏好品种/等级/规格/预算带/目的港/MOQ/采购周期。
- FR-13-03 供给对齐：可售库存批次（含临期急售）、期货合约、拍卖场次实时索引。
- FR-13-04 四维评分（0-100，权重可配置）：
  - **Matching Score**：需求画像与供给批次的属性相似度（规格/品种/等级/价格带/交期，向量+规则混合）；
  - **Opportunity Score**：预估成交额 × 成交概率（历史转化率）；
  - **Urgency Score**：临期库存、Buyer 断货周期、合约临近交割、库存积压天数；
  - **Profit Score**：预估佣金差 =（Buyer 可接受价 − 供应商底价）× 数量 × 费率。
- FR-13-05 商机对象（Opportunity）：`BY-xxx ↔ SP-xxx ↔ 批次` 三元组 + 四维分 + 推荐话术（AI 生成），推入 Broker 看板，红点/弹窗提醒，状态机（`NEW → CONTACTED → NEGOTIATING → WON / LOST`，LOST 记原因）。
- BR-13-01 P2 先实现可配置规则引擎（如"需求关键词 × 库存属性精确匹配 + 阈值"），行为数据积累 ≥3 个月后引入 embedding 相似度与转化率模型。

### M14 AI Broker（居间作业台）— P2

- FR-14-01 商机看板：M13 商机流 + 客户 360（脱敏）+ 供给面板三栏工作台。
- FR-14-02 一键外呼：点击代码旁"联系"→ Telephony 服务查密文号码发起双向桥接（Twilio Proxy），Broker 屏幕只见通话状态与时长；通话记录（时间/时长/商机 ID）入审计；录音留存策略遵循 GDPR（提示音 + 保留期配置）。
- FR-14-03 平台代发通知：Broker 编辑或 AI 生成话术 → 以"OUSSOURI 官方集采中心"名义发站内信/邮件（模板三语渲染），Broker 不可见收件人真实邮箱。
- FR-14-04 脱敏发单：调用 M12 一键发送检测报告/健康证副本。
- FR-14-05 居间代下单：双方口头达成一致 → Broker 录入价格与批次 → 生成"居间意向单"→ 系统向 Buyer 发带 Stripe 支付链接的通知（24h 内支付 20% 定金或全款锁货，联动 M05 预留 TTL）→ 支付即转正式订单入标准状态机。
- FR-14-06 AI Sales Copilot：基于商机上下文生成开场白、跟进邮件、议价建议（给出可让价空间 = Profit Score 反推）。
- BR-14-01 Broker 全程接触不到明文联系方式；导出功能对 Broker 角色全局禁用。

### M15 AI Translation（翻译中枢）— P2

- FR-15-01 翻译管道：业务内容（产品、等级、通知模板、单证字段）创建/变更时自动生成缺失语种译文（LLM），状态 `MACHINE_DRAFT → REVIEWED`，前台可展示草稿但带标记，人工复核后转正。
- FR-15-02 行业术语表（Glossary）：品种拉丁名、等级命名（如 Héritage Hezhe）、单证术语锁定翻译，注入 LLM prompt。
- FR-15-03 IM 会话实时翻译（P3）：跨语言沟通气泡内联翻译。
- BR-15-01 法律与合规文本（合同、发票、单证）机器翻译必须人工复核后方可使用。

### M16 Notification Center（通知中心）— P1

- FR-16-01 统一通知服务：站内信、邮件（平台域名发信）、短信/WhatsApp（预留通道抽象）；模板库三语，变量渲染。
- FR-16-02 事件驱动：订阅领域事件（订单迁移、证书到期、拍卖开始、商机产生、争议）自动触发；用户可配置偏好（渠道/频率/免打扰）。
- FR-16-03 发送记录与送达状态留痕；退信处理。
- BR-16-01 所有对外通知模板经 PII 过滤器；发件人一律为平台身份。

### M17 Internal Messaging（站内 IM）— P1

- FR-17-01 会话模型：Buyer↔平台、Supplier↔平台、Buyer↔Supplier（**必须挂靠业务对象**：产品/订单/RFQ，且经平台通道）；Broker 可加入任意会话作为居间方。
- FR-17-02 **敏感信息拦截**：消息发送前实时检测电话/邮箱/微信/WhatsApp/URL/银行账号（正则 + NER），命中即阻断发送、提示违规、写审计、累计触发风控（3 次冻结沟通权限并告警管理员）。
- FR-17-03 附件同样过滤（图片 OCR 检测联系方式，P3）；文件发送建议走 M12 脱敏通道。
- FR-17-04 消息已读回执、历史归档（争议证据链）。
- BR-17-01 IM 昵称一律显示平台代码；客服可见会话全文（争议处理），访问留审计。

### M18 Admin Console（管理后台）— P1

- FR-18-01 审核中心：供应商/采购商入驻、产品上架、拍卖申请、期货合约，统一待办流（多级审核可配置）。
- FR-18-02 配置中心：编码规则、佣金费率、可见性策略、状态机迁移表、单证清单模板、品类/品种/等级字典、汇率源、通知模板。
- FR-18-03 用户与 RBAC 管理：角色-权限矩阵（资源 × 操作 × 数据范围），内部账号强制 2FA。
- FR-18-04 内容管理：Banner、公告、行业资讯（三语）。
- BR-18-01 所有配置变更走审计 + 生效版本化（可回滚）。

### M19 Audit Center（审计中心）— P1

- FR-19-01 审计流水检索：按人/实体/操作类型/时间/IP 多维查询，导出（仅超管）。
- FR-19-02 **穿透审批（Access Escalation）**：查看真实身份字段需提交事由 → 按敏感级别即时放行（低敏，事后抄送）或审批后放行（高敏）→ 限时窗口（如 30 分钟）内可见 → 全程记录并实时抄送超管。
- FR-19-03 风控告警：异常模式检测（同一 Broker 高频穿透、IM 拦截高频触发、非工作时间批量查看）推送超管。
- BR-19-01 审计逻辑在服务层"写死"（代码级强制，不可被配置关闭）。

### M20 Analytics Dashboard（经营分析）— P3

- FR-20-01 平台看板：GMV、订单数、转化漏斗（浏览→加购→下单→支付）、品类/国别分布、佣金收入、Escrow 在途资金。
- FR-20-02 撮合业绩：Broker 人效（商机跟进量、成单率、佣金贡献）、商机漏斗。
- FR-20-03 供需缺口：需求信号（搜索/RFQ/LOI）与在售库存的品种×规格×国别差集热力图。
- BR-20-01 分析视图对角色分级（供应商只见自己数据的对标脱敏分位）。

### M21 Market Intelligence（市场情报）— P3

- FR-21-01 数据采集：HS Code 进出口贸易统计（UN Comtrade/Eurostat API）、公开价格源、行业新闻（RSS/爬虫，来源白名单）。
- FR-21-02 情报分析：国家维度进出口趋势、价格带变动、竞品动态、餐饮/分销/批发需求信号（结合站内行为数据）。
- FR-21-03 AI 自动报告：周报/月报（三语），LLM 基于结构化数据 + 新闻摘要生成，模板化章节，人工审核后可推送给订阅客户（增值服务）。
- BR-21-01 外部数据标注来源与抓取时间；AI 报告附数据引用，禁止无来源结论。

### M22 AI Assistant（全域 AI 助手）— P3（基础对话 P2 可先行）

- FR-22-01 角色化 Copilot：Buyer 助手（选品/合规问答/订单查询）、Supplier 助手（上架引导/单证准备清单）、Broker Copilot（见 M14）、清关助手（见 M11）、客服机器人（一线问答，转人工兜底）。
- FR-22-02 RAG 知识库：平台规则、品类知识（鲟鱼品种/等级/储存）、合规流程（CITES/报关）、FAQ 入 pgvector，多语检索。
- FR-22-03 工具调用：助手可查询用户自己权限内的数据（订单状态、物流轨迹），严格继承用户权限上下文（AI 不越权）。
- BR-22-01 LLM Provider 抽象层：默认 DeepSeek，按任务可路由（翻译/生成/推理可配不同模型）；所有输出过 PII 过滤器；Prompt 注入防护（用户输入隔离、工具白名单）。

## 4. 验收基线（Phase 1 MVP Definition of Done）

1. Guest 三语浏览脱敏市场；Buyer/Supplier 完成注册审核并登录。
2. 供应商上架一个带溯源批次与阶梯价的产品（经审核）。
3. Buyer 完成直购全流程：加购 → 下单 → Stripe 测试支付入 Escrow → 供应商确认发货 → Buyer 签收 → 自动分账（佣金截留）→ 双方全程只见对方平台代码。
4. IM 发送含手机号的消息被拦截并产生审计记录。
5. 管理员穿透查看某 Buyer 真实公司名：需填事由、留审计、超管收到通知。
6. 全流程中英法三语切换无硬编码文案。
